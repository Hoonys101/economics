### 3. ⚠️ Logic & Spec Gaps
*   **Escheatment Fallback Bypass (Wealth Orphanage Bug)**: 
    `EstateRegistry.process_estate_distribution` 메서드 내에서 상속자(Heir)에게 자금을 이체하는 `settlement_system.transfer()` 호출이 실패(예: 계좌 동결, 검증 실패 등으로 인해 `None`을 반환)하더라도, `distributed = True`가 무조건 실행되는 치명적인 논리적 오류가 있습니다.
    ```python
    tx = settlement_system.transfer(...)
    if tx:
        transactions.append(tx)
    distributed = True # BUG: tx가 None이어서 이체가 실패해도 무조건 True가 됨
    ```
    이 경우 상속자에게 돈이 가지 않았음에도 불구하고 하단의 국고 귀속(Escheatment) 로직이 스킵되어, 결국 망자의 계좌에 자금이 영구적으로 갇히는 **Wealth Orphanage** 현상이 그대로 발생하게 됩니다. `distributed = True`는 반드시 `if tx:` 블록 안으로 이동해야 합니다.

- **Title**: Post-Execution Hook Ghost Transactions
- **Symptom**: `EstateRegistry.process_estate_distribution` generates post-settlement transactions (inheritance/escheatment) that are executed by the ledger but not propagated up to the global `SimulationState.transactions` log.
- **Risk**: Financial flow is correct (M2/Zero-Sum maintained), but these transactions are invisible to macro-analytics and UI dashboards, creating a "Ghost Transaction" phenomenon where money moves without a visible trace in the main feed.
- **Solution**: Implement an `_internal_tx_queue` within `SettlementSystem`. Append side-effect transactions to this queue, and expose a `drain_internal_transactions()` method for the TickOrchestrator to append them to the global `SimulationState.transactions` at the end of each tick.
