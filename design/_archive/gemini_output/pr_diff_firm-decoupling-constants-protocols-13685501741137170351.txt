+# Firm Architecture Decoupling Report
+
+## 1. Architectural Insights
+
+### Constants Extraction
+To improve maintainability and decouple the Firm architecture from hardcoded values, we extracted key constants from `simulation/firms.py` into a new dedicated module `modules/firm/constants.py`. This centralization allows for easier configuration and tuning of firm behavior without modifying the core orchestrator logic.
+
+**Extracted Constants:**
+- `DEFAULT_MARKET_INSIGHT`: 0.5
+- `DEFAULT_MARKETING_BUDGET_RATE`: 0.05
+- `DEFAULT_LIQUIDATION_PRICE`: 1000
+- `INSIGHT_DECAY_RATE`: 0.001
+- `INSIGHT_BOOST_FACTOR`: 0.05
+- `INSIGHT_ERROR_THRESHOLD`: 1000.0
+- `DEFAULT_LABOR_WAGE`: 1000
+- `DEFAULT_SURVIVAL_COST`: 10.0
+- `DEFAULT_CORPORATE_TAX_RATE`: 0.2
+- `PRODUCTIVITY_DIVIDER`: 10.0
+- `DEFAULT_PRICE`: 1000
+
+### Protocol Purity
+We enforced Protocol Purity by introducing explicit protocols for `SalesEngine` and `BrandEngine` in `modules/firm/api.py`. The `Firm` orchestrator now relies on these protocols (`ISalesEngine`, `IBrandEngine`, `IFinanceEngine`, `IHREngine`) for type hinting, rather than concrete implementations. This decouples the orchestrator from specific engine implementations and facilitates future dependency injection or mocking.
+
+**New Protocols:**
+- `ISalesEngine`
+- `IBrandEngine`
+
+**Updated Protocols (Usage):**
+- `IFinanceEngine`
+- `IHREngine`
+
+### DTO Purity
+The new protocols utilize strict Data Transfer Objects (DTOs) for inputs and outputs, ensuring clear boundaries between the stateful orchestrator and stateless engines. We imported necessary DTOs like `SalesPostAskContextDTO` and `SalesMarketingContextDTO` into the API definition to support these protocols.
+
+## 2. Regression Analysis
+
+During the verification phase, we encountered significant failures in the `tests/integration/test_server_integration.py` and `tests/security/` suites. These failures were unrelated to the Firm logic refactoring but stemmed from missing dependencies in the test environment.
+
+**Issues Identified:**
+- `ModuleNotFoundError: No module named 'fastapi'`
+- `TypeError: object AsyncMock can't be used in 'await' expression` in `test_server_integration.py` and `test_websocket_auth.py`.
+
+**Root Cause:**
+The test environment lacked `fastapi`, `httpx`, and `websockets` libraries. Consequently, `tests/conftest.py` was falling back to mocking `websockets` with `MagicMock`, which caused `await ws.recv()` to fail because `MagicMock` is not awaitable.
+
+**Resolution:**
+We installed the required dependencies (`fastapi`, `httpx`, `pytest`, `pytest-asyncio`, `websockets`) using `pip`. This restored the correct behavior of the server integration tests, allowing them to run against the actual `websockets` library implementation rather than a broken mock.
+
+**Verification:**
+After installing the dependencies, all 925 tests passed, confirming that the architectural changes to `Firm` did not introduce any regressions and that the system integrity is maintained.
